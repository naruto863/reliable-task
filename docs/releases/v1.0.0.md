# ReliableTask v1.0.0

These release notes describe the v1.0.0 source/tag release. Maven Central publication remains a separate publish step that requires Central and GPG credentials.

## Highlights

- Freezes the core ReliableTask API/SPI for the first stable open-source line.
- Documents the v0.x to v1.0 upgrade path, including retained compatibility APIs and schema initialization strategy.
- Keeps the reliability model explicit: ReliableTask provides at-least-once execution, and business handlers remain responsible for idempotent external side effects.
- Prepares Maven Central publication with metadata, sources jars, javadoc jars, dry-run-safe release profile and guarded publish workflow.
- Upgrades the Spring Boot dependency-management baseline to 3.5.14 while keeping MyBatis-Plus unchanged.
- Adds opt-in JDK 21 virtual-thread execution through `reliable-task.executor.mode=virtual`; the default remains `platform`.
- Keeps `reliable-task-console` as an independent preview frontend, outside the Maven reactor and starter jars.
- Adds a v1.0 test matrix, example matrix, readiness report and open-source governance report.
- Adds CodeQL and Dependabot configuration for the repository governance path.
- Fixes Console dependency audit findings by upgrading the Vite/Vitest toolchain and verifying the Console still typechecks, tests and builds.

## Compatibility Notes

- No mandatory source-breaking or schema-breaking change has been identified for a normal `0.7.0` to `1.0.0` upgrade in the current protocol.
- Java 21 is the documented build/runtime baseline.
- Spring Boot dependency management is `3.5.14`.
- `reliable-task.executor.mode` defaults to `platform`; `virtual` is an opt-in execution mode that uses JDK 21 virtual threads and treats `max-size` as the concurrency limit.
- `TaskStore` remains as a compatibility facade. New code should prefer `TaskCommandStore`, `TaskQueryStore` or `TaskOperationsStore`.
- Deprecated `TaskSerializer` remains in v1.0 with `forRemoval=false`. New code should prefer `TaskPayloadSerializer` or `TaskPayloadCodec`.
- Existing task statuses and state-machine semantics remain stable.
- Admin APIs remain disabled by default. Admin writes remain opt-in and guarded by authorization, audit and confirmation.

## Getting Started

After v1.0.0 is visible on Maven Central, worker-only applications can use:

```xml
<dependency>
  <groupId>com.reliabletask</groupId>
  <artifactId>reliable-task-spring-boot-starter</artifactId>
  <version>1.0.0</version>
</dependency>
```

Applications that explicitly opt in to Admin REST APIs can add:

```xml
<dependency>
  <groupId>com.reliabletask</groupId>
  <artifactId>reliable-task-admin-spring-boot-starter</artifactId>
  <version>1.0.0</version>
</dependency>
```

Until Maven Central publication is separately confirmed, use a source build, local Maven installation or private Maven repository instead of claiming the Central coordinates are available.

## Upgrade Notes

Read the upgrade guide first:

- `docs/migration/v1.0-upgrade-guide.md`

Key points:

- Choose exactly one schema initialization path per database: `schema.sql`, Flyway or Liquibase.
- Verify application compatibility with Java 21 and Spring Boot 3.5.14 before production rollout.
- Keep the executor mode at the default `platform` unless you deliberately opt in to virtual threads and have reviewed handler blocking behavior.
- Validate a disposable database clone before production rollout.
- Do not remove retained compatibility APIs during a v1.0 upgrade.
- Keep task handlers idempotent for external systems such as payment, inventory, messaging or shipping.
- Do not put credentials, tokens, personal data or sensitive payloads into task payloads, diagnostics, idempotency keys or logs.

## Known Limits

- v1.0 does not provide exactly-once external side effects.
- v1.0 does not add official PostgreSQL, MongoDB, Redis, Kafka, RabbitMQ, RocketMQ or XXL-Job backends.
- v1.0 does not introduce DAG/workflow orchestration, Saga, approval flows or a multi-tenant RBAC platform.
- Console remains a preview operations UI and should be deployed only behind the same protected internal boundary as Admin APIs.
- Console/Admin roadmap and non-goals are documented in `docs/console-admin-roadmap.md`.
- Maven Central publish is not performed by the source/tag release and requires Central/GPG credentials plus the guarded publish workflow.

## Security

- Report vulnerabilities privately using `SECURITY.md`.
- Contributor onboarding and issue/PR routing are documented in `docs/community-onboarding.md`.
- Admin APIs must be protected by authentication, authorization, audit logging and network controls.
- Release workflow secrets must be provided through GitHub Actions secrets, never committed to the repository.
- The repository includes CodeQL and Dependabot configuration, plus a release-time security scan option.
