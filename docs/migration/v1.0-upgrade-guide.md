# ReliableTask v1.0 Upgrade Guide

- target_version: `v1.0.0`
- baseline_version: `0.7.0`
- api_compatibility: `docs/v1.0/RELIABLE_TASK_V10_API_COMPATIBILITY.md`
- schema_strategy: `docs/v1.0/RELIABLE_TASK_V10_SCHEMA_STRATEGY.md`
- created_at: 2026-06-14 +08:00

## 1. Upgrade Goal

This guide helps existing v0.x users prepare for ReliableTask v1.0.

ReliableTask v1.0 is intended to freeze the stable public API/SPI, database initialization strategy and release governance for the first stable open-source line. It keeps the database-backed at-least-once execution model. It does not introduce exactly-once external side effects, a new official database backend, a workflow engine, or automatic production database migration.

The v1.0 baseline is Java 21 and Spring Boot 3.5.14. Existing users should verify their application-level Spring Boot compatibility before switching production workloads.

Maven Central publication is part of the v1.0 release closure. Until that closure is complete, use a source build, local Maven installation, or a private Maven repository. After release closure, the same coordinates should use version `1.0.0` from Maven Central.

## 2. Quick Decision Matrix

| Current User State | Action Before v1.0 |
| --- | --- |
| New user evaluating ReliableTask | Start from the v1.0 README once release closure is complete; before that, use the preview source build and read this guide. |
| Existing v0.7.x user | Expect no mandatory schema change, but review Admin/Console boundaries and run the validation commands below. |
| Existing v0.6.x user | Confirm Worker/Admin starter split, Store SPI compatibility, payload codec migration and schema initialization path. |
| Existing v0.5.x or earlier user | Treat the upgrade as a staged migration: schema comparison, starter dependency review, Admin safety review and compatibility testing. |
| Production database with local DDL customizations | Validate a disposable clone first; ReliableTask v1.0 does not own private columns, triggers, indexes or archival jobs. |

## 3. Breaking Changes

No mandatory source-breaking or schema-breaking change has been identified for a normal v0.7.0 to v1.0.0 upgrade in the current v1.0 protocol.

The following historical 0.x changes still matter when upgrading from older versions:

| Area | Impact | Migration |
| --- | --- | --- |
| Admin starter split since v0.6 | Worker-only applications no longer receive Admin/Web dependencies from `reliable-task-spring-boot-starter`. | Add `reliable-task-admin-spring-boot-starter` only in applications that expose Admin REST APIs. |
| Admin write safeguards since v0.7 | Enabled writes still require write switch, authorization, audit and confirmation header before mutation. | Keep production Admin APIs behind internal network and auth controls; do not rely on frontend hiding. |
| Retry index naming since v0.6 | The retry table task-id index is `idx_retry_task_id`. | Compare existing schema before v1.0 rollout. |
| Console preview since v0.7 | The console is a separate frontend and not bundled into any Java starter. | Deploy `reliable-task-console/dist` independently behind the same protected operations boundary as Admin APIs. |
| Spring Boot baseline in v1.0 | Dependency management is upgraded to Spring Boot 3.5.14. | Run the validation commands below and verify application compatibility with Spring Framework 6.2.x-managed dependencies. |

If any later v1.0 task introduces an actual breaking change, it must update this section, `CHANGELOG.md`, readiness report and release notes before release closure.

## 4. Compatible Changes

The v1.0 compatibility policy retains the current core contracts:

- `TaskTemplate` submission flow remains the primary business API.
- `TaskHandler` remains the execution SPI.
- Task status codes and legal state transitions remain stable.
- Lightweight task events remain observability hooks, not durable event-sourcing records.
- `FailureClassifier`, retry strategies and exception formatting remain extension points.
- Worker starter remains worker-only.
- Admin starter remains explicit opt-in.

Runtime semantics stay at-least-once. Handlers and external systems must remain idempotent because retries, recovery and manual requeue can call business logic more than once.

## 5. Deprecated or Compatibility-Retained Surfaces

| Surface | v1.0 Status | Recommended Path |
| --- | --- | --- |
| `TaskSerializer` | Deprecated, retained, `forRemoval=false`. | Prefer `TaskPayloadSerializer`; use `TaskPayloadCodec` for context-aware payload handling. |
| `TaskStore` | Compatibility facade retained. | New store extensions should depend on `TaskCommandStore`, `TaskQueryStore` or `TaskOperationsStore` according to the narrow capability needed. |
| `reliable-task.serializer.type` | Bindable compatibility property; does not switch serializers. | Register a `TaskPayloadSerializer` bean or `TaskPayloadCodec` bean. |
| `reliable-task.store.table-prefix` | Bindable compatibility property; does not rewrite MyBatis table names. | Use the documented v1.0 table names. |
| `reliable-task.admin.port` and `reliable-task.admin.context-path` | Bindable compatibility properties; do not create a separate management server or remap Admin APIs. | Put Admin APIs behind application routing, reverse proxy and network controls. |

Do not delete compatibility-retained code during a v1.0 upgrade. Removal, if ever needed, must be planned as a later compatibility review.

## 6. Dependency and Module Steps

Before v1.0 is published to Maven Central:

```bash
mvn -B -DskipTests install
```

After v1.0 release closure, worker-only applications should use:

```xml
<dependency>
    <groupId>com.reliabletask</groupId>
    <artifactId>reliable-task-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

Applications that need Admin REST APIs must add the Admin starter explicitly:

```xml
<dependency>
    <groupId>com.reliabletask</groupId>
    <artifactId>reliable-task-admin-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

The console remains independent:

- it is not a Maven module in the Java reactor;
- it is not bundled into either Spring Boot starter;
- it should be built and deployed as static assets behind an internal operations boundary.

## 7. Configuration Review

Minimum production review:

| Configuration Area | v1.0 Guidance |
| --- | --- |
| `reliable-task.enabled` | Keep enabled only in applications that should run ReliableTask components. |
| `reliable-task.worker.enabled` | Enable for Worker applications; review batch size, max batch size and lock TTL. |
| `reliable-task.recovery.*` | Keep recovery enabled for reliability; validate timeout and max reset settings against business expectations. |
| `reliable-task.retry.*` | Review retry intervals, jitter and max delay before production rollout. |
| `reliable-task.executor.mode` | Defaults to `platform`. Use `virtual` only as an explicit Java 21 virtual-thread opt-in; `max-size` remains the concurrency limit while `core-size` and `queue-capacity` are compatibility settings. |
| `reliable-task.admin.enabled` | Disabled by default. Enable only in applications meant to expose Admin APIs. |
| `reliable-task.admin.write-enabled` | Disabled by default. If enabled, keep authorization, audit and confirmation requirements active. |
| `reliable-task.admin.console.payload-plaintext-enabled` | Disabled by default. Do not expose plaintext payload unless the payload is already safe for operators. |

Configuration changes should be tested in staging before changing production behavior. Do not use example or demo configuration as production defaults.

## 8. Schema Initialization Choices

Choose exactly one path per database:

| Path | Use When | File |
| --- | --- | --- |
| Direct SQL | Your deployment provisions schema manually. | `db/schema.sql` |
| Flyway | Your service already uses Flyway for database migrations. | `db/migration/V1__init_reliable_task_schema.sql` |
| Liquibase | Your service already uses Liquibase for database migrations. | `db/changelog/db.changelog-master.yaml` |

Do not mix these paths for the same ReliableTask tables. For example, if Flyway already owns the ReliableTask schema, do not also run `schema.sql` manually during startup.

For a normal v0.7.0 to v1.0.0 upgrade:

1. Verify that the existing ReliableTask database contains the six expected tables:
   `reliable_task`, `reliable_task_log`, `reliable_task_retry`, `reliable_task_worker`, `reliable_task_audit_log`, and `reliable_task_batch_operation`.
2. Compare indexes against the v1.0 baseline, especially `uk_biz_unique_key`, `idx_status_next_priority_id`, `idx_lock_expire`, `idx_retry_task_id`, and Admin audit/batch indexes.
3. Confirm there are no local private DDL changes that conflict with v1.0 reserved columns or indexes.
4. Upgrade ReliableTask artifacts in a staging environment.
5. Run default store tests and a real MySQL profile if the environment supports it.
6. Deploy the application artifact after the schema check is complete.

For v0.6.x users:

- The Flyway and Liquibase entry points already exist in the project line.
- Confirm the retry table index name is `idx_retry_task_id`.
- Keep `TaskStore` compatibility code in place; do not remove compatibility layers during the database upgrade.

For v0.5.x or earlier users:

- Compare your current schema with the v1.0 baseline before rollout.
- If tables were created by manual SQL and you now want Flyway or Liquibase to own them, baseline the migration tool only after confirming the existing schema is equivalent.
- Treat missing Admin, audit, worker or batch-operation tables as an environment-specific migration task that must be tested in a disposable clone first.

## 9. Validation Commands

Run from the repository root on Windows:

```powershell
cmd.exe /c mvn -B test
cmd.exe /c mvn -B -pl reliable-task-store -am test
cmd.exe /c mvn -B -Pmysql-it -pl reliable-task-store,reliable-task-executor -am test
```

If Docker/Testcontainers is unavailable, use a disposable local MySQL 8 test database:

```powershell
$env:RELIABLE_TASK_IT_JDBC_URL="jdbc:mysql://127.0.0.1:3306/reliable_task_it?useSSL=false&serverTimezone=UTC"
$env:RELIABLE_TASK_IT_USERNAME="reliable_task_it"
$env:RELIABLE_TASK_IT_PASSWORD="change-me"
cmd.exe /c mvn -B -Pmysql-local-it -pl reliable-task-store,reliable-task-executor -am test
```

Rules:

- A passing H2 MySQL-mode smoke test is useful but does not replace real MySQL release validation.
- If both Testcontainers and local MySQL are unavailable, record `BLOCKED_ENV`.
- Do not claim v1.0 real MySQL readiness until one real MySQL profile passes.
- Do not run MySQL integration tests against a production database.

## 10. Rollback Notes

If no schema change was applied, rollback is normally an application artifact rollback.

If local configuration changed Admin write behavior, rollback must also restore the previous Admin settings and network policy.

If a local team added private schema changes, rollback belongs to that team's migration plan. ReliableTask v1.0 does not own private columns, private indexes, custom triggers, or private archival jobs.

Do not drop ReliableTask tables during rollback unless the database is disposable and the data is no longer needed.

## 11. Release Checklist for Upgraders

Before adopting v1.0 in production:

- read `CHANGELOG.md` and `docs/releases/v1.0.0.md` once release notes exist;
- confirm Maven Central publication status or use an approved private artifact source;
- run Java tests and at least one real MySQL validation profile;
- confirm Admin and Console deployment boundaries;
- confirm handlers are idempotent under at-least-once execution;
- keep `TaskSerializer`, `TaskStore` and compatibility properties available until application code has migrated away from them;
- record any `BLOCKED_ENV` validation gaps in the release decision.
